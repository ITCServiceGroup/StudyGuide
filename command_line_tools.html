Command-Line Tools Study Guide

Command line tools are essential instruments in the network technician's toolkit, providing powerful methods for diagnosing, analyzing, and troubleshooting network connectivity issues. These tools offer direct access to network diagnostics and system information that may not be readily available through graphical interfaces. This guide covers fundamental command line utilities across different operating systems, including tools for tracing network paths, testing connectivity, managing DNS settings, and analyzing packet loss. By mastering these command line tools, technicians can more effectively identify network problems, measure performance metrics, and implement solutions across various platforms. Each section combines theoretical understanding with practical examples and common usage scenarios, enabling you to confidently utilize these tools in real-world troubleshooting situations.

1. Accessing Command-Line Interfaces

Different operating systems provide various ways to access their command-line interfaces. Understanding these access methods is crucial for network diagnostics and troubleshooting.

Windows

  • Press Windows + R, type cmd, press Enter
  • Search for "Command Prompt" in Start menu
  • Navigate to C:\Windows\System32 and run cmd.exe

macOS

  • Open Spotlight (Command + Space), type "Terminal"
  • Find Terminal in Applications > Utilities
  • Use Launchpad to locate Terminal

Chrome OS

  • Press Ctrl + Alt + T to open Crosh shell
  • Use Linux Terminal (if enabled) via Settings > Advanced > Developers
  • Click the Launcher, search for "Terminal", and open the app
Important: Some commands may require administrator privileges. On Windows, right-click Command Prompt and select "Run as administrator". On macOS/Linux, use sudo before commands requiring elevated privileges.

3. Command Usage and Examples

Traceroute / Tracert / Tracepath

Maps the path between your device and a destination, showing each hop and latency.

Windows: tracert google.com
Tracing route to google.com [142.250.xxx.xxx]
1 1 ms 1 ms 1 ms 192.168.1.1
2 5 ms 5 ms 5 ms 10.0.0.1
3 15 ms 14 ms 15 ms 172.16.0.1
Mac: traceroute google.com
traceroute to google.com (142.250.xxx.xxx), 64 hops max
1 192.168.1.1 1.234 ms 1.123 ms 1.234 ms
2 10.0.0.1 5.678 ms 5.567 ms 5.789 ms
3 172.16.0.1 15.123 ms 14.567 ms 15.789 ms
Mac (MTR): sudo mtr -n google.com
Host Loss% Snt Last Avg Best Wrst StDev
1. 192.168.1.1 0.0% 10 0.5 0.6 0.4 0.8 0.1
2. 10.0.0.1 0.0% 10 5.4 5.5 5.3 5.8 0.2
3. 172.16.0.1 0.0% 10 15.3 15.4 15.2 15.9 0.3
Chrome OS: tracepath google.com
1?: [LOCALHOST] pmtu 1500
1: 192.168.1.1 0.123ms
2: 10.0.0.1 5.456ms
3: 172.16.0.1 15.789ms
Pro Tip: Use traceroute to identify where network slowdowns or failures occur. High latency or timeouts at a specific hop indicate a problem at that point in the network.

Ping and Packet Loss

Test connectivity and measure packet loss. Always use at least 50 pings for accuracy.

Windows: ping -n 50 8.8.8.8
Reply from 8.8.8.8: bytes=32 time=15ms TTL=57
...
Packets: Sent = 50, Received = 50, Lost = 0 (0% loss)
Mac/Chrome OS: ping -c 50 8.8.8.8
50 packets transmitted, 50 received, 0% packet loss, time 49000ms
Pro Tip: Any packet loss above 1% may indicate a network issue. Consistent loss or high latency should be investigated.

DNS Troubleshooting

Resolve domain name resolution issues by managing and refreshing your system's DNS cache settings.

Windows: ipconfig /flushdns
Mac: sudo killall -HUP mDNSResponder
Chrome OS: Clear browser cache (DNS managed by Chrome)
When to clear DNS cache: After DNS server changes, when websites don't load, or after network config changes.

Scenario-Based Troubleshooting

  • Slow websites: Test with ping and traceroute. If DNS is set manually, clear DNS cache and set to automatic.
  • Device-specific WiFi issues: Forget and reconnect to WiFi, or factory reset router if needed.
  • Intermittent drops: Check for firmware updates, analyze WiFi channel congestion, and check for interference or overheating.
  • Packet loss: Use ping with 50+ packets. Investigate high loss with traceroute and check for physical issues.

4. DNS Cache Management

DNS cache can become corrupted or outdated, leading to connectivity issues. Knowing how to clear the DNS cache is essential for troubleshooting.

DNS Cache Clearing Commands:

Operating System Command Notes

macOS sudo killall -HUP mDNSResponder Password required
Chrome OS Clear browser cache DNS managed by Chrome
When to Clear DNS Cache:
  • After DNS server changes
  • When websites don't load correctly
  • After network configuration changes
  • If security issues are suspected

5. Interpreting Command Output

Understanding command output is crucial for effective troubleshooting. Each command provides specific patterns and indicators that help diagnose network issues.

Common Output Patterns

Pattern Meaning Severity Action Required
Request timed out No response received within expected time Medium Check connectivity, firewall rules, and network congestion
Destination host unreachable No valid route to target High Verify network configuration, gateway settings, and routing tables
TTL expired in transit Packet took too many hops Medium Investigate routing paths, possible routing loops
General failure Network adapter or driver issue High Check network adapter status, update drivers
Connection refused Target actively rejected connection High Verify service status, port availability, and firewall rules

Traceroute Analysis

traceroute to google.com (172.217.xxx.xxx), 30 hops max
 1  192.168.1.1  1.123 ms  *  1.234 ms
 2  10.0.0.1  15.678 ms  !X  15.789 ms
 3  * * *
  • * - Timeout or no response from hop
  • !X - Communication administratively prohibited
  • !H - Host unreachable
  • !N - Network unreachable
  • Multiple responses per hop indicate multiple paths
  • Increasing times suggest congestion or distance

IP Configuration Analysis

Ethernet adapter Local Area Connection:
   IP Address. . . . . . . . : 192.168.1.100
   Subnet Mask . . . . . . . : 255.255.255.0
   Default Gateway . . . . . : 192.168.1.1
  • Media disconnected - Physical connection issue
  • 169.254.x.x - APIPA address, DHCP failure
  • 0.0.0.0 - No IP configuration
  • Multiple gateway entries may indicate routing issues

DNS Resolution Analysis

Server:   192.168.1.1
Address:  192.168.1.1#53

Non-authoritative answer:
Name: example.com
Address: 93.184.216.34
  • Non-authoritative answer - Response from cache or recursive resolver
  • NXDOMAIN - Domain does not exist
  • SERVFAIL - DNS server failure
  • REFUSED - DNS server declined to answer

6. Best Practices

Security Best Practices

  • Elevated Privileges:
    • Use admin/root access only when necessary
    • Verify commands before running with elevated privileges
    • Always revert to standard user privileges after completion
  • Sensitive Information:
    • Redact sensitive data before sharing logs
    • Avoid exposing internal IP schemes in public documentation
    • Use secure channels when sharing diagnostic data
  • Remote Troubleshooting:
    • Use encrypted connections for remote access
    • Implement timeouts for automated tasks
    • Log all remote diagnostic sessions

Performance Testing Guidelines

  • Baseline Testing:
    • Establish performance baselines during normal operation
    • Document typical response times and patterns
    • Test during both peak and off-peak hours
  • Test Parameters:
    • Use consistent test durations and intervals
    • Test with appropriate sample sizes (minimum 50 pings)
    • Include varied packet sizes in testing
  • Network Conditions:
    • Document network load during tests
    • Account for bandwidth limitations
    • Consider environmental factors

Automation and Scripting

  • Script Development:
    • Include error handling in automated scripts
    • Add appropriate timeouts and retry logic
    • Implement logging for automated processes
  • Output Management:
    • Use structured output formats (CSV, JSON)
    • Implement log rotation for long-running tests
    • Include timestamps in all logs

Basic Troubleshooting Workflow

  1. Start with basic connectivity tests (ping)
  2. Progress to more detailed tests (traceroute)
  3. Document all command outputs
  4. Test from multiple locations when possible
  5. Use appropriate sample sizes for tests
  6. Check both internal and external IPs
  7. Clear DNS cache when needed
  8. Analyze patterns in test results

System Resource Management

  • Resource Monitoring:
    • Monitor CPU and memory usage during tests
    • Check network interface utilization
    • Watch for resource exhaustion
  • Parallel Operations:
    • Limit concurrent network tests
    • Space out automated operations
    • Consider system load when scheduling tests

Cross-Platform Considerations

  • Command Compatibility:
    • Verify command syntax across platforms
    • Use platform-specific flags appropriately
    • Document OS-specific variations
  • Output Differences:
    • Account for varying output formats
    • Handle different line endings
    • Consider character encoding differences
Documentation Requirements:
  • Test Environment:
    • Date and time of test
    • Operating system and version
    • Network conditions
    • System resource state
  • Command Details:
    • Complete command syntax used
    • Command parameters and flags
    • Expected vs actual results
    • Any error messages or warnings
  • Follow-up:
    • Actions taken based on results
    • Resolution status
    • Recommendations for future testing
